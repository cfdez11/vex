# VexJS Monorepo

This repository contains the VexJS framework and related packages.

## Packages

| Package | Description |
|---|---|
| [`vexjs`](./vexjs) | The VexJS framework — file-based routing, SSR/CSR/SSG/ISR, Vue-like reactivity |
| [`demo`](./demo) | Example project using VexJS |
| [`vscode-extension`](./vscode-extension) | VS Code extension with syntax highlighting for `.vex` files |

## Getting Started

### Monorepo (development)

```bash
pnpm install
pnpm --filter demo dev
```

### New project (from npm)

Install the package:

```bash
npm install @cfdez11/vex
```

Then set up your `package.json`. The `vex` CLI handles all framework commands (`dev`, `build`, `build:static`, `start`). CSS tooling (e.g. Tailwind) is added separately:

```json
{
  "name": "my-app",
  "type": "module",
  "scripts": {
    "dev": "run-p dev:server dev:css",
    "dev:server": "vex dev",
    "dev:css": "npx @tailwindcss/cli -i ./src/input.css -o ./public/styles.css --watch",
    "build": "vex build",
    "build:static": "vex build:static",
    "build:css": "npx @tailwindcss/cli -i ./src/input.css -o ./public/styles.css --minify",
    "start": "vex start"
  },
  "dependencies": {
    "@cfdez11/vex": "^0.6.0"
  },
  "devDependencies": {
    "npm-run-all": "^4.1.5",
    "tailwindcss": "^4.0.0"
  }
}
```

> CSS is optional — if you're not using Tailwind, replace `dev` with just `"vex dev"` and remove the css scripts.

Copy `.env.example` to `.env` and set your environment variables:

```bash
# .env
VEX_PORT=3001         # port for the dev/prod server (default: 3001)
```

#### Available CLI commands

| Command | Description |
|---|---|
| `vex dev` | Start the development server with HMR |
| `vex build` | Prebuild: generate component bundles and route files |
| `vex build:static` | Full static build → `dist/` folder (deployable to any static host) |
| `vex start` | Start the production server (requires `vex build` first) |

## Client Bundling with esbuild

VexJS uses **esbuild** to compile and bundle the `<script client>` blocks of every `.vex` component.

### Why esbuild?

Before esbuild, client bundles were generated by string concatenation: the framework collected import statements, rewrote their paths manually, and joined everything into a single JS string. This worked for the framework's own imports (`vex/reactive`, `vex/html`) but had two hard limitations:

1. **npm packages could not be used in `<script client>`.** Bare specifiers like `import _ from 'lodash'` were passed through unchanged — the browser has no module resolver and would throw a runtime error.
2. **User utility files were not bundled.** Files imported via `@/utils/counter` were served individually at runtime by a dedicated Express route (`/_vexjs/user/*`) that rewrote their imports on the fly. This meant an extra HTTP round-trip per utility file on every page load, and it made static builds incomplete.

esbuild eliminates both limitations at build time, with no changes needed in user code.

### What esbuild handles

| Import type | Before | After |
|---|---|---|
| `vex/reactive`, `vex/html` | Manually rewritten to `/_vexjs/services/...` | Resolved by alias plugin, marked external |
| `@/utils/counter` (project alias) | Served at runtime via `/_vexjs/user/*` | Bundled inline into the component |
| `./helpers` (relative) | Served at runtime via `/_vexjs/user/*` | Bundled inline via `resolveDir` |
| `lodash`, `date-fns` (npm) | Left unresolved — browser error | Resolved from `node_modules` and bundled |

### Framework singletons stay external

`reactive.js`, `html.js` and the other framework runtime files at `/_vexjs/services/` are marked **external** in the esbuild config. This means esbuild leaves their import statements intact in the output; the browser fetches them from the static middleware at runtime.

This is intentional: all components on a page must share the **same instance** of `reactive.js`. If esbuild inlined it, each component bundle would carry its own copy — reactive state would not propagate across components and the reactivity system would break.

### The alias plugin

The esbuild alias plugin (`vexjs/server/utils/esbuild-plugin.js`) teaches esbuild about VexJS-specific import conventions:

- `vex/*` and `.app/*` → rewritten to `/_vexjs/services/...` and marked external
- `@/*` → resolved to the project `SRC_DIR` root (bundled)
- Relative imports and npm packages → handled by esbuild's default resolver

### Using npm packages in components

With esbuild in place, any npm package installed in the project can be imported directly in a `<script client>` block:

```html
<script client>
  import confetti from 'canvas-confetti';
  import { format } from 'date-fns';

  // works — esbuild resolves and bundles these at build time
</script>
```
